# UAPI 协议转换层严格规范验证与修复任务

## 目标

你正在对 UAPI 项目的协议转换层进行严格、逐节、逐字段、逐行为的审查和修复。

本任务有两个核心目标：

1. **验证规范是否准确**

   * 检查 `UAPI_PROTOCOL_CONVERSION_LAYER_IMPLEMENTATION_SPEC.md` 中每一节的描述是否与 `upstream/` 下的源码事实一致。
   * `upstream/` 包含所有规范验证所需的外部事实来源，包括 Bifrost、官方客户端源码、竞品源码和协议参考实现。
   * 如果规范描述与 `upstream/` 源码不一致，必须记录差异并修正规范。

2. **验证并修复 UAPI 实现**

   * 检查 `internal/` 下的 UAPI 实现是否符合已经确认准确的规范。
   * 如果实现缺失、不完整或边界行为错误，必须修复。
   * 修复后必须补充或运行测试验证。

# 一、关键原则：严格区分规范验证与实现验证

本任务包含两个完全不同的阶段，必须严格区分。

## 阶段 A：验证并修正规范文档

本阶段的目标是验证：

```text
UAPI_PROTOCOL_CONVERSION_LAYER_IMPLEMENTATION_SPEC.md
```

是否准确描述了 `upstream/` 中的协议行为、官方客户端行为、竞品实现行为和 Bifrost 行为。

**阶段 A 只能读取以下来源作为事实依据：**

```text
upstream/bifrost/
upstream/cliproxyapi/
upstream/cockpit-tool/
upstream/antigravity-manager/
upstream/codex/
upstream/gemini-cli/
upstream/claude-code-source-*
```

以及 `upstream/` 下其他与当前章节明确相关的源码文件。

**阶段 A 禁止把以下内容作为规范准确性的依据：**

```text
internal/
UAPI 实现代码
UAPI 自己的更新文档
UAPI 自己的测试
UAPI 当前行为
```

也就是说：

* 验证“规范是否准确”时，不能看 `internal/` 来判断规范对不对。
* `internal/` 只能在阶段 B 使用。
* 如果 `upstream/` 与规范冲突，优先相信 `upstream/`，并修正规范。
* 如果 `internal/` 与 `upstream/` 冲突，不能修改规范去迎合 `internal/`，而应在阶段 B 修复 `internal/`。

## 阶段 B：验证并修复 UAPI 实现

只有在当前章节的规范已经被阶段 A 确认准确或完成修正后，才能进入阶段 B。

本阶段才允许读取：

```text
internal/
```

并验证 UAPI 实现是否符合“已经确认准确的规范”。

判断顺序必须是：

```text
upstream/ 源码事实
        ↓
修正并确认规范
        ↓
internal/ UAPI 实现验证
        ↓
修复 internal/ 实现或补充 unsupported 行为
        ↓
运行测试
```

禁止出现以下错误流程：

```text
读取 internal/ → 推断规范是否正确
读取 UAPI 文档 → 推断 upstream 行为
按当前 UAPI 实现反向修改规范
```

# 七、每节必须执行的详细流程

## 0. 验证顺序约束

每节必须先验证规范本身是否准确，再验证 UAPI 实现。

严格顺序如下：

1. 读取当前章节完整规范。
2. 提取当前章节全部规范点。
3. 根据 `upstream/` 源码逐项核对规范描述。
4. 如果规范与 `upstream/` 源码不一致，优先相信 `upstream/`，并修正规范文件。
5. 只有在规范已被确认准确或已修正后，才进入 UAPI 实现验证。
6. UAPI 实现验证只以已确认准确的规范为准。
7. 如果 UAPI 未实现、部分实现或实现质量不合格，必须修复到符合规范，或明确标记为 unsupported。
8. 修复后运行相关测试。

注意：

```text
规范准确性验证读取的是 upstream/ 源码，不是 UAPI 源码，也不是 UAPI 的源码更新文档。
UAPI 源码只用于后续验证 internal/ 实现是否符合已确认准确的规范。
```

## 1. 读取当前章节

从 `UAPI_PROTOCOL_CONVERSION_LAYER_IMPLEMENTATION_SPEC.md` 读取当前章节完整内容。

必须明确当前处理的是：

```text
第 X 节: [章节标题]
```

## 2. 批量提取本节验证对象

从当前章节中一次性提取全部关键对象，包括但不限于：

* 协议
* route
* endpoint
* method
* request 字段
* response 字段
* streaming event
* error shape
* status code
* header
* usage 字段
* cache token 字段
* native field
* raw body 行为
* loss 行为
* unsupported 行为
* parser / emitter / converter 函数
* same-protocol 行为
* cross-protocol 行为
* async / large payload / files / batch 边界

## 3. 批量验证规范准确性

对当前章节中的每个规范点或规范点组，使用 `upstream/` 源码批量验证。

判断标准：

* 源码存在且行为一致：`✅准确`
* 源码存在但描述不完整：`⚠️部分准确`
* 源码与规范冲突：`❌不准确`
* 规范没有源码依据：`⚠️缺少依据`

如果规范不准确，必须修改规范文件，而不是只在报告里说明。

## 4. 批量验证 UAPI 实现

在规范已确认准确或已修正后，才允许进入 UAPI 实现验证。

在 `internal/` 目录中查找对应实现，重点包括：

```text
internal/relay/provider/convert/
internal/relay/provider/ir/
internal/relay/provider/stream/
internal/relay/provider/openai/
internal/relay/provider/anthropic/
internal/relay/provider/gemini/
internal/relay/provider/antigravity/
internal/relay/
```

验证内容包括：

* 请求解析是否正确
* 请求发射是否正确
* 响应解析是否正确
* 响应发射是否正确
* same-protocol 是否保持原生结构
* cross-protocol 是否经过 IR
* raw body 是否在允许场景中保留
* native fields 是否保留或记录 loss
* unsupported 行为是否明确拒绝
* streaming event 是否完整转换
* error / status / header 是否符合矩阵
* usage / cache token 是否正确提取
* 文件、大 payload、batch、async 边界是否正确处理

判断标准：

* 完整符合规范：`✅已修复`
* 完全缺失：`❌未修复`
* 部分实现但有遗漏：`⚠️部分修复`
* 实现存在但质量不合格：`⚠️需调整`

## 5. 决策并执行

根据验证结果做出明确决策：

* 规范错误：更新规范
* UAPI 未实现：开始修复
* UAPI 部分实现：补齐缺口
* UAPI 实现质量不足：调整实现
* 规范和实现均正确：接受当前状态
* 上游明确不支持且 UAPI 不应支持：标记 unsupported

不要只报告问题。

除非遇到无法自行决策的架构冲突，否则必须直接修复。

## 6. 测试验证

每节处理完成后，必须运行相关测试。

优先运行局部测试：

```bash
go test ./internal/relay/provider/convert/... -v -count=1
go test ./internal/relay/provider/stream/... -v -count=1
go test ./internal/relay/provider/antigravity/... -v -count=1
```

如果修改影响范围较大，运行更广泛测试：

```bash
go test ./internal/relay/... -v -count=1
go test ./internal/... -v -count=1
```

如果测试失败，必须继续修复，直到相关测试通过，或明确说明阻塞原因。
