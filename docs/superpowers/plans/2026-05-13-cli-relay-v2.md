## Phase 2: 数据层 (PostgreSQL + GORM)

**目标**: 定义全部 7 张表，实现连接管理与加密字段。

### 2.1 模型定义

| 表名 | 主键 | 说明 |
|------|------|------|
| `channels` | UUID v7 | 渠道/号池分组 |
| `accounts` | UUID v7 | 上游账号，credentials 加密存储 |
| `tokens` | UUID v7 | 下游 API Key (sk-xxx) |
| `plans` | UUID v7 | 计费方案 |
| `token_plans` | UUID v7 | Token-Plan 绑定关系 |
| `logs` | BIGSERIAL | 请求日志，按月分区 |
| `audit_log` | BIGSERIAL | 审计日志，按月分区 |

### 2.2 关键实现点
- `accounts.credentials` 使用 `crypto` 包的 AES-256-GCM 加密/解密
- `token_plans.window_usage` 和 `window_reset_at` 使用 `jsonb` 类型
- GORM 自动迁移 (`AutoMigrate`) 用于开发，生产使用 `migrations/`
- 连接池配置: `max_open_conns=25`, `max_idle_conns=5`, `conn_max_lifetime=5m`

## Phase 3: fasthttp 服务器与路由

**目标**: 单端口服务器，三条路径分流。

### 3.1 路由规则

```
GET  /               → 静态文件 (Web UI index.html)
POST /api/admin/*    → 管理 API (JWT 鉴权)
ANY  /v1/*           → 转发引擎 (Bearer sk-xxx 鉴权)
```

### 3.2 fasthttp Server 配置
- `ReadTimeout: 60s`, `WriteTimeout: 60s`
- `MaxIdleConnDuration: 30s`, `MaxConnDuration: 300s`
- `MaxConnsPerHost: 200`
- `ConnPoolStrategy: FIFO`
- `RetryIfErr: StaleConnectionRetryIfErr` (处理 io.EOF/断连重试)
- `ReadBufferSize: 64KB`

### 3.3 中间件
- CORS (管理 API)
- 请求日志 (仅管理 API)
- 安全响应头

## Phase 4: 转发引擎核心

### 4.1 Adaptor 接口

```go
type Adaptor interface {
    Init(channel *Channel, account *Account)
    GetRequestURL(path string) (string, error)
    SetupRequestHeader(h *fasthttp.RequestHeader)
    ConvertRequest(body []byte) ([]byte, error)
    ParseUsage(respBody []byte) (promptTokens, completionTokens int, err error)
    GetChannelName() string
}
```

### 4.2 转发流水线 (单次请求)

```
① 提取 Bearer Token → 查 tokens 表验证
② 解析请求体中的 model 字段
③ model → 渠道类型映射 (models 表)
④ 从该渠道的号池 PickAccount() (平滑加权轮询)
⑤ 获取账号的 Adaptor (根据 channel.type)
⑥ Adaptor.ConvertRequest() 转换格式
⑦ Adaptor.SetupRequestHeader() 注入上游认证
⑧ fasthttp 发送请求到上游
⑨ 接收响应:
   - 非流式: 直接返回，ParseUsage 统计
   - 流式: SSEStreamReader 零拷贝转发
⑩ 计费: PreConsume → 响应后 Settle/Refund
⑪ 写日志 (异步批量写入)
```

### 4.3 SSE 流式转发

借鉴 Bifrost 的 SSEStreamReader:

```go
type SSEStreamReader struct {
    eventCh chan []byte    // buf=1，单事件流水线
    closeCh chan struct{}
    closeOnce sync.Once
    current []byte
}
```

-  producer goroutine 从上游 fasthttp response body 逐行读取 SSE event
-  每读到一个完整 event，调用 `SendEvent("", data)` 写入 channel
-  fasthttp `SetBodyStream(reader, -1)` 消费 channel
-  客户端断开时 `Close()` 关闭 closeCh，producer 感知后退出

### 4.4 失败自动切换

```go
// 上游请求失败 (5xx/连接错误/超时)
if err != nil {
    // 1. 标记当前账号冷却 (weight=0, cooldown_until=now+5min)
    pool.CooldownAccount(accountID, 5*time.Minute)
    
    // 2. 如果号池还有其他可用账号，重试
    if retryAccount, ok := pool.PickAccount(); ok {
        goto ④  // 重新选号转发
    }
    
    // 3. 号池耗尽，返回 503
    return errorNoAvailableAccount
}
```

## Phase 5: 号池管理 (平滑加权轮询)

### 5.1 数据结构

```go
type WeightedAccount struct {
    Account        *Account
    Weight         int   // 有效权重 (冷却时为 0)
    CurrentWeight  int   // 调度临时权重
    OriginalWeight int   // 配置权重 (冷却恢复用)
}

type AccountPool struct {
    mu           sync.RWMutex
    accounts     []WeightedAccount  // copy-on-write
    totalWeight  int
    channelID    uuid.UUID
}
```

### 5.2 平滑加权轮询算法

```go
func (p *AccountPool) Pick() *Account {
    p.mu.RLock()
    defer p.mu.RUnlock()
    
    var best *WeightedAccount
    total := 0
    for i := range p.accounts {
        p.accounts[i].CurrentWeight += p.accounts[i].Weight
        total += p.accounts[i].Weight
        if best == nil || p.accounts[i].CurrentWeight > best.CurrentWeight {
            best = &p.accounts[i]
        }
    }
    if best != nil {
        best.CurrentWeight -= total
        return best.Account
    }
    return nil
}
```

### 5.3 冷却与恢复

- **冷却**: `Weight = 0`，`cooldown_until = time.Now().Add(duration)`
- **恢复**: 定时任务(每 30s)扫描 `cooldown_until < now` 的账号，恢复 `Weight = OriginalWeight`
- **热更新**: 管理端修改账号后，通过 `sync/atomic` swap 新的 `[]WeightedAccount` 切片 (copy-on-write)

## Phase 6: 双模式计费系统

### 6.1 计数模式 (count_based)

```json
// plans.limits
[
  {"window": "1h", "max": 100},
  {"window": "5h", "max": 500},
  {"window": "7d", "max": 1200}
]
```

- 窗口格式: `数字+单位` (h=小时, d=天, w=周)
- `token_plans.window_usage`: `{"1h": 45, "5h": 200, "7d": 800}`
- `token_plans.window_reset_at`: `{"1h": "2026-05-13T14:00:00Z", ...}`
- 每次请求前检查：当前时间 > reset_at ? usage=0 : usage+1
- 检查窗口是否超限，超限返回 429

### 6.2 Token 模式 (token_based)

```json
// plans.model_ratios
{
  "gpt-4o": 1.0,
  "claude-3-5-sonnet": 1.2,
  "gemini-1.5-pro": 0.8
}
```

- `preConsume`: 根据 `max_tokens` 或预估 token 数，预扣 `token_quota`
- `settle`: 实际返回 usage 后，计算真实消耗 = `prompt_tokens + completion_tokens * completion_ratio * model_ratio`
- delta = actual - preConsumed，多退少补
- `refund`: 请求失败时异步退回预扣额度

### 6.3 并发安全

- 使用 `SELECT FOR UPDATE` 行锁锁定 `token_plans` 记录
- 或 GORM 的 `Clauses(clause.Locking{Strength: "UPDATE"})`

## Phase 9: 安全与审计

### 9.1 加密
- `accounts.credentials`: AES-256-GCM，密钥从环境变量 `ENCRYPTION_KEY` 读取
- 管理密码: bcrypt (cost=12)

### 9.2 JWT
- 签发: `HS256`，过期时间 24h
- Payload: `sub`(adminID), `iat`, `exp`

### 9.3 审计日志
- 记录所有管理 API 的变更操作 (who, what, when, old_val, new_val)
- 异步写入 `audit_log` 表

### 9.4 数据清理
- 软删除: `deleted_at` 字段
- 管理表: 定时任务每天清理 `deleted_at < now - 30d` 的记录
- 日志表: 按月分区，6个月后 `DROP PARTITION`

```go

    cfg := config.Load("config.yaml")
    db := db.Init(cfg.Database)
    crypto.Init(cfg.Security.EncryptionKey)

    server := server.New(cfg, db, poolManager, billing)
```
